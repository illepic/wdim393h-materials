Let's Talk Native vs Non-Native

<img src="images/hatin.jpg">

Browsers implement <img> natively, with full "understanding" of the object being embedded. Attributes like height, width, file path and file size are all available to the browser to manipulate at will.

Non-Native

The <object> tag hands off all responsibility to an external program or plugin

<object width="425" height="344">
  <param name="movie" value="http://www.youtube.com/video" />
  <param name="allowFullScreen" value="true" />
  <param name="allowscriptaccess" value="always" />
  <embed src="http://www.youtube.com/video"
  type="application/x-shockwave-flash" allowscriptaccess="always"
  allowfullscreen="true" width="425" height="344">
  </embed>
</object>

Why this sucks

Handing off rendering of multimedia is web standards "cheating".

  • The code is complicated to write and understand
  • Browser has to verify that the user has a third party plugin
  • Or has the knowledge and ability to download it.
  • Plugins frequently cause instability and crash
  • Freak out less tech-savvy people by prompting to download something to view the desirable content.
  • Ever try to overlay an element (css dropdown menus, overlay lightboxes) overtop of an <object>?

<video>,<audio>

HTML5 provides a standardized way to play video directly in the browser, with no plugins required.

Markup

Markup is similar to <img>, but with a few handy attributes.

<audio src="elvis.ogg" controls preload="auto"></audio>

<video src="videofile.ogv" controls preload="meta">
  Download the <a href=videofile.ogv>HTML5 video</a>
</video>

The "Stack"

This is a "safe" stack of <source> elements: 

<video>
  <source src="video/wall.webm" type='video/webm; codecs="vp8, vorbis"' />
  <source src="video/wall.mp4" type='video/mp4; codecs="avc1.42E01E, mp4a.40.2"' />
  <source src="video/wall.theora.ogv" type='video/ogg; codecs="theora, vorbis"' />
  Video tag not supported. Download the video <a href="movie.webm">here</a>.
</video>

We included multiple <source> elements to "fall back" for browsers that don't support certain formats. At this early stage it is also recommended to include detailed "type" attributes to let browsers best decide rendering.

Attributes

Both <audio> and <video> share the following attributes:

  • src:

    The URL to play.

  • autoplay:

    Plays on load

  • loop:

    Replay the file after completion

  • controls:

    Implement browser/OS default player controls

Attributes

Both <audio> and <video> share the following attributes:

  • preload: "buffer" the file automatically before some kind of user interaction
    • auto: let the browser decide to buffer or not based on current conditions
    • none: do not preload
    • meta: tells the browser that it should prefetch metadata (sizing, first frame, poster, duration etc) but that it shouldnt download anything until the user activates the controls
  • poster (<video> only): a link to an image to show as a placeholder

Gotchas

  • Firefox has a bug with "loop". jQuery fix: 
    $('video[loop]').bind('ended', function () {
	this.play();
});
  • Firefox will not recognize video without proper Apache config serving the correct MIME type. Add the following to httpd.conf: 
    AddType video/ogg .ogv
AddType video/mp4 .mp4
AddType video/webm .webm
  • Poster image will set width/height by default. (See demo)
  • iPads require mp4 being the first <source> in the video stack.
